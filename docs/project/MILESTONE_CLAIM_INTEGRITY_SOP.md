# Milestone Claim Integrity SOP

## 1. Purpose

This SOP prevents a milestone from claiming a learner-visible, runtime, rendered, PDF, production, human-review, or D0 result that is not supported by the exact execution path and artifact evidence.

The following are explicitly not equivalent:

```text
data structure ready != runtime integrated
candidate text authored != worksheet changed
standalone preview != production-equivalent renderer output
CI green != milestone claim correct
review packet exists != production review is ready
```

Every new task must submit a machine-readable claim manifest under:

```text
data/project/milestones/<TASK_ID>.claim.json
```

The repository-wide validator and required workflow fail closed.

## 2. Evidence levels

| Level | Name | Maximum supported claim |
|---|---|---|
| E0 | `PLANNING_ONLY` | planning and scope only |
| E1 | `DATA_STRUCTURE_READY` | schema, registry, mapping, candidate structure |
| E2 | `CONTENT_AUTHORED` | new content exists outside the production-equivalent path |
| E3 | `SHADOW_RUNTIME_INTEGRATED` | production resolver/generator logic is exercised in an isolated mode |
| E4 | `PRODUCTION_EQUIVALENT_OUTPUT_VERIFIED` | production renderer plus HTML/PDF output and visible difference evidence |
| E5 | `PRODUCTION_ADMITTED` | approved production registry and selector admission |
| E6 | `D0_COMPLETE` | stable generator, validator, renderer and worksheet output |

A task may target a higher level than it currently reaches. `actualEvidenceLevel` controls every allowed claim.

## 3. Required task-start declaration

Before implementation begins, the task contract must identify:

```text
EXPECTED_USER_VISIBLE_CHANGE
TARGET_RUNTIME_PATH
TARGET_GENERATOR_PATH
TARGET_RENDERER_PATH
TARGET_HTML_ARTIFACT
TARGET_PDF_ARTIFACT
HUMAN_REVIEW_TYPE
FINAL_ACCEPTANCE_SURFACE
```

A content-change task cannot begin without a target runtime/generator path. A PDF-change task cannot begin without a target renderer and PDF artifact.

## 4. Claim manifest

The manifest must validate against:

```text
data/project/schema/milestone-claim-evidence.schema.json
```

It records:

- target and actual evidence levels;
- individual claims;
- exact test and artifact paths;
- human-review type and authority;
- goal distance before/after;
- the next task and its prerequisite level.

Every listed evidence path must exist. Production-equivalent review artifacts must also own verified SHA-256 hashes.

## 5. Visible result gate

A task that claims visible output change must provide all of the following:

```text
runtimeIntegrated = true
productionEquivalentGeneratorUsed = true
productionRendererUsed = true
htmlOutputVerified = true
pdfOutputVerified = true
visibleOutputChanged = true
beforeAfterEvidencePaths.length > 0
htmlArtifactPaths.length > 0
pdfArtifactPaths.length > 0
```

The following cannot be used as production-output evidence:

- registry counts;
- candidate counts;
- binding metadata;
- fixtures;
- manually authored preview HTML;
- output generated by a renderer different from the production renderer.

## 6. Human-review types

### 6.1 Draft content review

`draft_content_review` may review candidate wording, naturalness, age appropriateness, or semantic intent at E2.

It must declare:

```text
canUnlockProduction = false
```

It cannot authorize runtime integration, production admission, worksheet acceptance, PDF acceptance, or D0.

### 6.2 Production-equivalent output review

`production_equivalent_output_review` may begin only at E4, after the exact production-equivalent resolver, generator, renderer, HTML, and PDF artifacts have passed.

The review must reference immutable artifact hashes. A separate hand-authored preview is not an admissible substitute.

## 7. Goal-distance integrity

Evidence levels constrain the closest allowed distance:

| Actual level | Closest allowed distance |
|---|---|
| E0 | D4 |
| E1 | D3 |
| E2 | D2 |
| E3 | D1 |
| E4 | D1 |
| E5 | D1 |
| E6 | D0 |

A task cannot reduce the distance beyond its actual evidence level.

## 8. Next-step integrity

The next task cannot require a prerequisite level higher than the current actual level. This prevents jumps such as:

```text
E2 candidate text
→ production admission
```

The correct sequence is:

```text
E2 content authored
→ E3 shadow runtime
→ E4 production-equivalent HTML/PDF
→ production-equivalent human review
→ E5 production admission
→ E6 D0
```

## 9. Required CI failures

The validator emits blocking codes including:

```text
MCI_CLAIM_MANIFEST_MISSING
MCI_CLAIM_EXCEEDS_EVIDENCE
MCI_RUNTIME_FALSE_BUT_OUTPUT_CLAIMED
MCI_RENDERER_FALSE_BUT_PDF_CLAIMED
MCI_VISIBLE_DIFF_EVIDENCE_MISSING
MCI_HUMAN_REVIEW_ARTIFACT_MISSING
MCI_HUMAN_REVIEW_TYPE_AMBIGUOUS
MCI_PRODUCTION_REVIEW_BEFORE_E4
MCI_NEXT_STEP_SKIPS_REQUIRED_LEVEL
MCI_DISTANCE_REDUCTION_UNSUPPORTED
MCI_D0_WITHOUT_FULL_PIPELINE
MCI_PREVIEW_USED_AS_PRODUCTION_EVIDENCE
MCI_LEGACY_OUTPUT_UNCHANGED
MCI_PR_BODY_MANIFEST_MISMATCH
```

Any blocking code means:

```text
CI = FAIL
PR = BLOCKED
MERGE = FORBIDDEN
```

## 10. PR body contract

Every task PR must include these exact fields:

```text
Milestone Claim Manifest: `data/project/milestones/<TASK_ID>.claim.json`
Actual Evidence Level: `E?_*`
Maximum Claim: `...`
Visible Output Changed: `true|false`
Human Review Type: `none|draft_content_review|production_equivalent_output_review`
Human Review Ready: `true|false`
```

The required workflow compares these values with the changed manifest.

## 11. Development-error policy

Development defects may be found and corrected on a branch. The unacceptable condition is a false completion claim merged to `main` and discovered later from a worksheet or PDF by the operator.

Project-level targets:

```text
POST_MERGE_FALSE_COMPLETION_CLAIMS = 0
USER_DISCOVERED_PIPELINE_DISCONNECTS = 0
HUMAN_REVIEW_WITHOUT_REVIEWABLE_ARTIFACT = 0
D0_CLAIMS_WITHOUT_FULL_PIPELINE = 0
```

## 12. P12 precedent

`GCTX-P12` is the mandatory negative precedent. It authored five candidate contexts but did not change the production resolver, renderer, or PDF. Its correct level is E2, not production-review-ready.

The governance test suite must continue to prove that the former combination below is blocked:

```text
contentAuthored = true
runtimeIntegrated = false
productionRendererUsed = false
pdfOutputVerified = false
visibleOutputChanged = false
humanReviewReady = true
```
