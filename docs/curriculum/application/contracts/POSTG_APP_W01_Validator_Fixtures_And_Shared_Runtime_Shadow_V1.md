```text

PRODUCTION_ADMISSION_CHANGE = false
```

## 1. Purpose

A03 executes the first shared runtime over the Wave 01 A01/A02 artifacts without changing the public generator.

```text
A01 context-bound single-application candidates
+ A02 N+1 and PBL blueprints
→ deterministic shadow fixtures
→ shared validator runtime
→ positive PASS evidence
→ negative fail-closed evidence
→ Wave 01 runtime readback
```

The runtime verifies that candidate lineage, mathematical-role coverage, context chain, answer meaning, N+1 interpretation evidence and PBL dependencies can be consumed together.

## 2. Evidence boundary

A03 is a shadow validation milestone. It does not claim that every Canonical Operation Model has a production numeric generator.

Each fixture records:

```text
fixtureAdapterId
operationFamily
roleValues
computedAnswerPayload
answerRole
answerUnitCandidate
context lineage
interpretation evidence
expected validation result
```

The numeric witness is generated by a deterministic operation-family adapter explicitly linked to an A00 operation-family candidate. The exact production PatternSpec generator remains authoritative downstream.

## 3. Assessment scope versus runtime scope

Wave 01 retains all 15 Golden units in A00 assessment authority.

The A03 runtime executes only units with at least one A01 application candidate:

```text
15 Golden units assessed
→ suitable KnowledgePoints receive A01 candidates and A03 runtime fixtures
→ units with no suitable application KnowledgePoint remain excluded
→ excluded units must not receive forced story fixtures
```

Therefore:

```text
Golden assessment coverage = 15 units
Application runtime coverage = exact distinct sourceIds represented by A01 candidates
Excluded application units = assessment units minus eligible runtime units
```

Runtime coverage must equal eligible application coverage exactly. It must not be inflated by creating stories for `APPLICATION_NOT_APPLICABLE` KnowledgePoints.

## 4. Fixture coverage

For every A01 suitable candidate:

```text
1 positive fixture
1 wrong-answer-role negative fixture
1 unit-mismatch negative fixture
```

For every A02 N+1 proof blueprint:

```text
1 calculation-pass / interpretation-fail fixture
1 counterfactual interpretation fixture
```

For every A02 PBL blueprint:

```text
1 valid dependency-graph fixture
1 broken-dependency negative fixture
```

All fixtures are deterministic and reproducible.

## 5. Operation-family shadow adapters

The adapter is selected only from the A00 `operationFamilyCandidates` attached to the KnowledgePoint. Adapter output cannot grant production admission.

## 6. Shared validator layers

```text
V1 Candidate lineage validator
V2 M01 hierarchy and surface validator
V3 Role coverage validator
V4 Shadow numeric witness validator
V5 Answer role and unit validator
V6 Interpretation witness validator
V7 N+1 diagnostic validator
V8 PBL dependency and final synthesis validator
V9 Production admission guard
```

A negative fixture must fail with the intended error class and must not pass due to an unrelated earlier error.

## 7. Diagnostic separation

A03 preserves:

```text
CALCULATION_PASS_INTERPRETATION_FAIL
```

The corresponding fixture keeps the numeric witness valid and removes only the interpretation evidence. The validator must report interpretation failure while keeping `calculationPass = true`.

## 8. PBL shadow execution

PBL fixtures execute graph-level logic only:

```text
all non-first tasks consume prior milestones or shared constraints
one final task exists
final task consumes at least two milestones
all referenced milestones exist
counterfactual propagation reaches the final decision
```

A03 does not generate student-facing PBL pages.
